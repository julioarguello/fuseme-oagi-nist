## [fuseme/mariadb-10.3-import] - 2026-03-17

Branch `fuseme/mariadb-10.3-import` off `master`. Goal: run the OAGIS 10.3 CC table
import on MariaDB instead of Oracle.

### Why this branch exists

The upstream OAGi SRT codebase targets **Oracle 12c**. We need the 39 CC tables populated
to generate OpenAPI 3.1 specs from OAGIS data. Since we do not have access to an Oracle
instance, all changes adapt the import pipeline to run on **MariaDB 10.6**.

### Detailed rationale

#### Change 1: Spring Boot 1.4.2 -> 1.5.22

**This is NOT a MariaDB-related change.** It fixes a Hibernate bug.

| Property | Original | Updated |
| :--- | :--- | :--- |
| `spring.boot.version` | `1.4.2.RELEASE` | `1.5.22.RELEASE` |
| `spring.framework.version` | `4.3.4.RELEASE` | `4.3.25.RELEASE` |
| `spring.data.jpa.version` | `1.10.5.RELEASE` | `1.11.23.RELEASE` |
| `spring.data.commons.version` | `1.12.5.RELEASE` | `1.13.23.RELEASE` |
| `spring.data.rest.version` | `2.5.5.RELEASE` | `2.6.11.RELEASE` |

**Root cause**: Spring Boot 1.4.2 ships Hibernate 5.0.12. This version has a bug
([HHH-11410](https://hibernate.atlassian.net/browse/HHH-11410)) where converters with
`@Converter(autoApply=true)` are registered twice when both `@EntityScan` and classpath
scanning find the same converter class. This causes:

```
Caused by: org.hibernate.AssertionFailure: AttributeConverterDefinition already registered
```

Spring Boot 1.5.x ships Hibernate 5.2.x which fixes this. The upgrade stays within the
1.x train (no API breaking changes) and uses the final patch release (1.5.22) for maximum
stability.

**Why not just remove `autoApply`?** We did both. The upgrade is still needed because
Spring Boot 1.4.2's `@EntityScan` has a secondary issue: its import path
(`org.springframework.boot.orm.jpa.EntityScan`) was deprecated and relocated to
`org.springframework.boot.autoconfigure.domain.EntityScan` in 1.4.x, but the compiled
bytecode in 1.4.2 can still trigger the old behavior. Using 1.5.22 ensures the new
location is the canonical one.

#### Change 6: Removed `@Converter(autoApply=true)` -- Safety analysis

**This change is safe because all entity fields already have explicit `@Convert` annotations.**

The `autoApply=true` flag tells JPA "apply this converter to ALL fields of this type
automatically." However, every entity in the codebase already declares the converter
explicitly via `@Convert(converter = XyzConverter.class)`. When both mechanisms are present,
`@Convert` takes precedence -- `autoApply` is redundant.

Proof -- all 19 field-level `@Convert` annotations:

| Converter | Entity.field |
| :--- | :--- |
| `CoreComponentStateConverter` | `AggregateCoreComponent.state`, `AssociationCoreComponent.state`, `BasicCoreComponent.state`, `BasicCoreComponentProperty.state`, `AssociationCoreComponentProperty.state`, `DataType.state` |
| `RevisionActionConverter` | `AggregateCoreComponent.revisionAction`, `AssociationCoreComponent.revisionAction`, `BasicCoreComponent.revisionAction`, `BasicCoreComponentProperty.revisionAction`, `AssociationCoreComponentProperty.revisionAction`, `DataType.revisionAction` |
| `BasicCoreComponentEntityTypeConverter` | `BasicCoreComponent.entityType` |
| `OagisComponentTypeConverter` | `AggregateCoreComponent.oagisComponentType` |
| `DependencyTypeConverter` | `ModuleDep.dependencyType` |
| `ABIEStateConverter` | `AggregateBusinessInformationEntity.state`, `ProfileBOD.state`, `TopLevelAbie.state` |

**No entity field uses these enum types without a `@Convert` annotation.** Removing
`autoApply` therefore changes nothing at runtime.

#### Change 5: `application.yml` -- `globally_quoted_identifiers`

MariaDB has reserved words that Oracle does not (e.g., `release`). The `release` column in
the `acc` table causes SQL syntax errors unless the column name is quoted. Setting
`globally_quoted_identifiers: true` wraps all identifiers in backticks.

#### Change 9: `DataImportScriptPrinter` -- `Number` cast

The original code casts `COUNT(*)` results to `BigDecimal` (Oracle's JDBC driver returns
this type). MariaDB's JDBC driver returns `BigInteger`. Using `Number` (the common
superclass) makes the code database-agnostic.

### How to run the import

Use the orchestration script:

```bash
./scripts/srt-import.sh          # full import (idempotent)
./scripts/srt-import.sh --reset  # drop + recreate database first
```
